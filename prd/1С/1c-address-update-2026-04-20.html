Изменения по адресу в payload для 1С

Дата: 20 апреля 2026. Проект: avterminal-api. Поток: amoCRM → 1С.

payload update address field 1C integration

1. Что поменяли сегодня

В payload для физлица поле полного адреса переименовано из Address в address.

  • Причина: привести имя поля к тому же стилю, что и остальные поля payload.
  • Было: поле выбивалось из общего формата, потому что начиналось с заглавной буквы.
  • Стало: теперь используется строчное имя address.

Итог: payload стал консистентным, а адресное поле теперь называется так же предсказуемо, как email, inn, snils и остальные ключи.

2. Было → стало

Было

{
  "clientType": "individual",
  "client": {
    "registrationAddress": "г. Владивосток, ул. Светланская, д. 1",
    "Address": "690066, Приморский край, Ленинский, Владивосток, пос. Трудовое, Светланская, 1, 12"
  }
}

Стало

{
  "clientType": "individual",
  "client": {
    "registrationAddress": "г. Владивосток, ул. Светланская, д. 1",
    "address": "690066, Приморский край, Ленинский, Владивосток, пос. Трудовое, Светланская, 1, 12"
  }
}

3. Почему адресов два

В payload для физлица сейчас участвуют два разных адресных поля. Они не дублируют друг друга и нужны для разных задач.

Поле в payload Что означает Откуда берется Как выглядит
registrationAddress Адрес регистрации одной строкой Из отдельного поля amoCRM, в первую очередь по field_id=808697, дальше через поиск по названию поля и коду ADDRESS г. Владивосток, ул. Светланская, д. 1
address Полный составной адрес из деталей контакта Собирается вручную из нескольких полей: индекс, субъект, район, город, населенный пункт, улица, дом, квартира 690066, Приморский край, Ленинский, Владивосток, пос. Трудовое, Светланская, 1, 12

Коротко: registrationAddress — это готовая строка адреса регистрации. address — это более полный адрес, который API конструирует из адресных компонентов контакта.

4. Как формируется поле address

Поле address строится только для физлица и собирается по частям в коде метода buildContactAddress().

address = индекс + субъект федерации + район + город + населенный пункт + улица + дом + квартира Дальше: 1. из каждого поля берется значение; 2. пустые и null-значения отбрасываются; 3. оставшиеся части склеиваются через ", ".

Это значит, что:

  • если часть адреса не заполнена, она просто не попадет в строку;
  • лишние запятые не появятся, потому что пустые элементы удаляются до склейки;
  • если не заполнено вообще ничего, поле address вернется как null.

5. Как формируется registrationAddress

Поле registrationAddress собирается по другому правилу. Здесь API сначала пытается взять уже готовое значение из выделенного кастомного поля amoCRM.

  • Приоритет 1: поле с field_id=808697.
  • Приоритет 2: поиск по названиям вроде адрес регистрации, регистрац, пропис, адрес.
  • Приоритет 3: fallback через поле с кодом ADDRESS.

Иными словами, registrationAddress не собирается по кускам. Это попытка взять уже существующий адрес регистрации в готовом текстовом виде.

6. Почему это важно для 1С

  • registrationAddress удобно использовать как адрес прописки, если 1С ожидает именно регистрационный адрес одной строкой.
  • address полезен там, где нужен максимально полный почтовый или фактический адрес из составных реквизитов контакта.
  • Наличие двух полей позволяет не терять ни "официальный" адрес регистрации, ни детализированный адрес из адресных компонентов.

Важно: после сегодняшнего обновления интеграции 1С нужно читать поле client.address, а не client.Address.

7. Рекомендуемый пример payload для физлица

{
  "clientType": "individual",
  "client": {
    "fullName": "Иванов Иван Иванович",
    "phone": "+79990001122",
    "email": "ivan@example.com",
    "passportSeries": "0505",
    "passportNumber": "123456",
    "passportIssuedBy": "УМВД России",
    "passportIssueDate": "2026-04-10",
    "passportDepartmentCode": "250-001",
    "registrationAddress": "г. Владивосток, ул. Светланская, д. 1",
    "address": "690066, Приморский край, Ленинский, Владивосток, пос. Трудовое, Светланская, 1, 12",
    "inn": "123456789012",
    "snils": "123-456-789 00",
    "dealerName": "ООО Ромашка Авто",
    "dealerInn": "1234567890"
  }
}

В этом примере registrationAddress и address намеренно разные: первое поле хранит адрес регистрации как готовую строку, второе — полный адрес из составных частей.
