# План разработки: интеграция amoCRM <-> 1С (v2)

## 1. Основание плана

- Финальное ТЗ `v2` по интеграции amoCRM и 1С.
- Структура текущего amoCRM-аккаунта из отчета `avterminal-amocrm-account-report.html`.
- Текущая кодовая база в `/Applications/XAMPP/xamppfiles/htdocs/avterminal/avterminal-api`.

Важно: в финальном `v2` основная область работ сужена до двух потоков:

1. `amoCRM -> 1С`: передача данных для создания/поиска контрагента.
2. `1С -> amoCRM`: прием факта оплаты и автоматический перевод сделки на этап `Оплачено`.

Поток `amoCRM -> 1С` по созданию счетов в `v2` исключен и в этот план не входит.

## 2. Что уже есть в проекте

В `avterminal-api` уже реализован каркас для интеграций с внешними системами через amoCRM:

- `AmoCRMService` для инициализации SDK клиента.
- `LeadService` для поиска сделки по VIN и обновления статусов/полей.
- `CustomFieldService` для записи значений в кастомные поля.
- HTTP endpoints в `routes/api.php`.
- Логирование через Laravel/Telescope.

Это значит, что новую доработку лучше строить не с нуля, а как отдельный модуль рядом с текущими `dt-status` и `td-status`.

## 3. Целевой scope разработки

### 3.1. Поток A: amoCRM -> 1С, синхронизация контрагента

Триггер:

- перевод сделки в этап `Договор` основной воронки `7523034 / 62360726`.

Система должна:

1. Проверить, что у сделки заполнен обязательный набор данных.
2. Собрать payload из сделки и связанной сущности клиента.
3. Передать payload в 1С через согласованный интеграционный канал.
4. Принять ответ от 1С с идентификатором найденного/созданного контрагента.
5. Сохранить этот идентификатор в amoCRM для повторных обращений и защиты от дублей.
6. Обеспечить безопасную повторную отправку по кнопке/ручному действию без создания дублей.

### 3.2. Поток B: 1С -> amoCRM, факт оплаты

Триггер:

- входящий запрос от 1С после фиксации оплаты.

Система должна:

1. Принять payload об оплате.
2. Найти сделку в amoCRM по согласованному ключу сопоставления.
3. Обновить технические поля интеграции и лог оплаты.
4. Проверить, что выполнено условие "все обязательные к оплате позиции оплачены".
5. Если условие выполнено, перевести сделку на этап `Оплачено` (`64577710`).
6. Если условие не выполнено, сохранить факт частичной оплаты без перевода этапа.

## 4. Данные и маппинг

### 4.1. Данные, которые уже есть в amoCRM

Из отчета подтверждены нужные поля:

- Сделка:

    - `Прохождение лаборатории для получения СБКТС и выдачи ЭПТС` (`916205`)
    - `Хранение на СВХ` (`916207`)
    - `Стоимость перевозки (клиент)` (`982363`)
- Контакт:
  - `Телефон`
  - `Кем выдан`
  - `Код подразделения`
  - `Место регистрации`
  - `Серия паспорта`
  - `Номер паспорта`
  - `СНИЛС`
  - `ИНН`
  - раздельные поля `Фамилия` / `Имя` / `Отчество`
- Компания:
  - `ИНН`
  - `Телефон`
  - карточка компании как отдельная сущность в аккаунте присутствует.

### 4.2. Что нужно учесть в коде

- Текущее `LeadService` ориентировано в основном на сделку и первый контакт.
- Для интеграции с 1С потребуется явная поддержка двух сценариев:
  - клиент = физлицо -> брать контакт;
  - клиент = юрлицо -> брать компанию, а не только контакт.
- Нужен отдельный слой нормализации данных перед отправкой в 1С:
  - чистка телефона;
  - сборка ФИО;
  - нормализация даты;
  - приведение пустых значений;
  - выбор источника ИНН.

## 5. Требуемые изменения в backend

### 5.1. Новые конфиги

Добавить в `config/amocrm.php` и, при необходимости, отдельный `config/integration_1c.php`:

- ID этапа-триггера `Договор`.
- ID этапа назначения `Оплачено`.
- ID технических полей 1С.
- список обязательных полей для отправки контрагента;
- правила расчета "все оплачено";
- параметры транспорта:
  - URL 1С;
  - токен / логин / пароль / сертификат;
  - таймауты;
  - режим sandbox/test.

### 5.2. Новые сервисы

Предлагаемая структура:

- `app/Services/Integration1C/CounterpartyPayloadBuilder.php`
- `app/Services/Integration1C/CounterpartySyncService.php`
- `app/Services/Integration1C/PaymentSyncService.php`
- `app/Services/Integration1C/PaymentStateResolver.php`
- `app/Services/Integration1C/OneCClient.php`

Функции:

- сбор payload для 1С;
- отправка в 1С;
- обработка ответов 1С;
- запись техполей в amoCRM;
- перевод сделки по этапам;
- идемпотентность входящих событий.

### 5.3. Новые endpoints

Минимально:

- `POST /api/1c/counterparty/sync`
  - ручной/служебный вызов на повторную отправку сделки;
- `POST /api/1c/payment-status`
  - входящий webhook из 1С по оплате;
- опционально `POST /api/1c/counterparty/callback`
  - если 1С отвечает асинхронно.

Если обмен будет идти не webhook-ом, а через файловую папку/опрос 1С, вместо части endpoints нужен job/command для выгрузки и загрузки файлов.

### 5.4. Триггер отправки из amoCRM

Нужно выбрать один из вариантов:

1. amoCRM Digital Pipeline / webhook на изменение этапа.
2. Наш polling/job, который отслеживает сделки на этапе `Договор` и флаг "не отправлено".
3. Ручная кнопка в виджете amoCRM, которая вызывает backend.

Для первой итерации safest-вариант:

- webhook/pipeline trigger на этап `Договор`;
- плюс ручной endpoint для повторной отправки.

## 6. Техническая логика потоков

### 6.1. Поток A: sync контрагента

Шаги:

1. Получить сделку по ID.
2. Подтянуть контакт и, при необходимости, компанию.
3. Определить тип клиента по полю `Вид клиента`.
4. Провалидировать обязательные поля.
5. Сформировать payload для 1С.
6. Отправить запрос в 1С.
7. Сохранить в amoCRM:
   - `1C_COUNTERPARTY_ID`;
   - дату последней синхронизации;
   - статус синхронизации;
   - текст последней ошибки, если есть.
8. Записать лог в Telescope.

### 6.2. Поток B: факт оплаты

Шаги:

1. Принять payload от 1С.
2. Проверить подпись/токен/источник.
3. Идемпотентно обработать событие оплаты.
4. Найти сделку:
   - по `1C_COUNTERPARTY_ID`, либо
   - по VIN, либо
   - по номеру сделки/договора, если это будет согласовано.
5. Обновить технические поля оплаты.
6. Сравнить оплаченные позиции с ожидаемым набором платных услуг в сделке.
7. При полном соответствии перевести сделку на этап `Оплачено`.

## 7. Что именно нужно доработать в amoCRM

Новые техполя, которые желательно завести заранее:

- `ID контрагента в 1С`
- `Статус синхронизации с 1С`
- `Дата последней синхронизации с 1С`
- `Ошибка синхронизации с 1С`
- `Дата последней оплаты из 1С`
- `Статус оплаты из 1С / агрегированный статус`
- `Лог оплат 1С` или `Последний payload 1С`
- `Номер договора` как отдельное поле в сделке, если он действительно нужен в обмене

Опционально:

- checkbox `Отправлено в 1С`
- button/action для `Повторно отправить в 1С`

## 8. Проверки и тестирование

### 8.1. Unit / feature

- построение payload для физлица;
- построение payload для юрлица;
- валидация обязательных полей;
- обработка ответа 1С с новым контрагентом;
- обработка ответа 1С с найденным контрагентом по ИНН;
- частичная оплата;
- полная оплата;
- защита от повторного входящего события;
- отказ 1С / таймаут / невалидный ответ.

### 8.2. Интеграционные сценарии

- перевод сделки в `Договор` -> успешная синхронизация;
- повторная отправка без дубля;
- оплата одной услуги без перевода в `Оплачено`;
- оплата всех услуг с переводом в `Оплачено`;
- ошибочный payload от 1С;
- отсутствие обязательных данных в amoCRM.

## 10. Основные риски

- в ТЗ не зафиксирован окончательный payload между нашей стороной и 1С;
- не определен точный ключ сопоставления оплаты со сделкой;
- не описано, как именно определяется "все счета оплачены", если поток создания счетов из amoCRM отменен;
- для юрлица в текущем коде нет отдельной логики работы с компанией;
- поле `Номер договора` в amoCRM по отчету явно не найдено;
- не согласована архитектура обмена: webhook, polling, файловая папка, callback.

## 11. Что нужно утвердить до старта разработки

1. Финальный транспорт обмена с 1С.
2. Формат request/response для обоих потоков.
3. Набор обязательных полей для физлица и юрлица.
4. Источник ИНН для физлица и юрлица.
5. Правило, по которому 1С сообщает об оплате.
6. Правило, по которому backend считает сделку полностью оплаченной.
7. Список и ID новых технических полей в amoCRM.
8. Нужен ли в payload `Номер договора`, и откуда его брать.

## 11.1. Зафиксированные решения (2026-03-16)

- Поле в amoCRM для `ID контрагента в 1С` уже создано в `lead`: `986431`.
- Статус синхронизации, дата/время синхронизации и текст ошибки:
  - не выносим в отдельные поля;
  - пишем в комментарии к лиду (примечания/notes).
- `last_payload_hash` храним на сервере (SQLite), не в amoCRM.
- `Номер договора` в текущей итерации не используем.
- Техническую привязку к 1С ведем на уровне `lead` (не `contact/company`).
- Тип `ИНН` в `companies` оставляем как есть (`numeric`) без изменений в amoCRM.

## 12. Результат разработки

После реализации система должна:

- автоматически передавать контрагента из amoCRM в 1С при этапе `Договор`;
- сохранять идентификатор контрагента 1С обратно в amoCRM;
- принимать уведомления об оплате из 1С;
- переводить сделку на этап `Оплачено` только при выполнении согласованного условия полной оплаты;
- поддерживать безопасную повторную отправку без дублей.
