Суперподробный мануал по интеграции с 1С

Проект: avterminal-api, поток интеграции контрагентов amoCRM ↔ 1С.

Актуально по коду: март 2026 Маршруты: /api/* (без /api/v1) Модель обмена: Buffer + Pull + Callback

0. Что это за документ

Это практический мануал для человека, который не в контексте проекта. Он отвечает на три вопроса:

  1. Как интеграция работает по шагам (включая алгоритмы и статусы).
  2. Какие именно поля и откуда берутся.
  3. Как воспроизвести и протестировать работу end-to-end, чтобы дать предметную обратную связь.
Важно: документ отражает фактическую реализацию в коде. Если что-то расходится с ранними ТЗ/планами, ориентируйтесь на этот мануал и текущий код.

1. Архитектура и границы

1.1 Реально работающий поток

Сейчас реализован поток A: передача данных контрагента из amoCRM в 1С через буфер:

amoCRM событие (стадия "Договор") / ручной триггер │ ▼ POST /api/amocrm/deals/contract-ready?secret=... │ ▼ Сбор данных сделки+контакта/компании в payload │ ▼ Запись в onec_counterparty_buffers (status=pending) │ ▼ 1С периодически делает GET /api/amocrm/integrations/1c/contacts/pending │ ▼ Запись помечается как pulled │ ▼ 1С обрабатывает и шлет POST /api/amocrm/integrations/1c/contacts/result │ ▼ Запись получает status=processed/error (+ при успехе пишется 1cId в кастомное поле лида amoCRM)

1.2 Что пока не реализовано

  • Отдельный webhook оплаты /integrations/1c/payments/webhook в коде отсутствует.
  • Массовый callback (массив объектов) в /contacts/result не поддерживается, только одиночный объект.

Секрет для contract-ready

Секрет хранится в ONEC_AMO_WEBHOOK_SECRET.

Сервис принимает его из любого источника (по приоритету):

  1. secret в body ($request->input('secret')).
  2. secret в query.
  3. header X-Amo-Webhook-Secret.
  4. header X-Webhook-Secret.
Если secret не задан в конфиге, contract-ready вернет 500 с ошибкой "Webhook secret is not configured".

3. Поток A: contract-ready (amoCRM → буфер)

3.1 Форматы входного payload

Метод принимает 3 формата (выявляются автоматически):

Источник Минимальный фрагмент source в ответе
JSON интеграции {"dealId": 123456} json_deal_id
amoCRM кнопка {"lead":{"id":123456,"pipeline_id":...,"status_id":...}} amo_button
amoCRM status webhook {"leads":{"status":[{"id":123456,"pipeline_id":...,"status_id":...}]}} amo_status_webhook
При одновременном наличии нескольких форматов "побеждает" последний проверенный в коде (leads.status[0]).

3.2 Фильтр по стадии "Договор"

Проверяются pipeline_id и status_id события.

  • Ожидаемые значения берутся из config('amocrm.onec.*').
  • По умолчанию: pipeline=7523034, status=62360726.
  • Если pipeline_id/status_id не пришли, событие не блокируется и обработка продолжается.
  • Если пришли, но не совпали, возвращается 200 и {"status":"ignored","reason":"event is not for contract stage"}.

3.3 Сбор payload из amoCRM

Сервис CounterpartyFlowService::buildPayloadFromLead() делает:

  1. Запрашивает лид по leadId с расширениями contacts, companies.
  2. Берет первый контакт/компанию, если они привязаны.
  3. Считывает custom fields по фиксированным ID.
  4. Определяет тип клиента:
    • если в поле типа клиента есть подстрока "юр" → clientType=legal,
    • иначе clientType=individual.
  5. Собирает финальный payload и позже добавляет в него requestId.

Формат payload в буфере/выдаче pending:

{
  "vin": "XTA210...",
  "dealId": 25147637,
  "clientType": "individual | legal",
  "clientTypeRaw": "сырой текст поля из amoCRM",
  "client": { "... зависит от clientType ..." },
  "deal": {
    "dealNumber": "25147637"
  },
  "_meta": {
    "contactId": 29639483,
    "companyId": 29639633
  },
  "requestId": "req_ab12cd34ef56"
}

3.4 Алгоритм дедупликации

Дедупликация выполняется только по паре (lead_id, payload_hash).

  1. Считается payload_hash = sha256(json(payload_without_requestId)).
  2. В таблице ищется любая запись этой сделки с таким hash (даже если она уже processed или error).
  3. Если найдена:
    • новая запись не создается,
    • возвращается {"status":"already_buffered","requestId":"..."},
    • в сделку добавляется сервисный комментарий.
  4. Если не найдена:
    • генерируется новый requestId вида req_ + 12 random chars,
    • создается запись pending в буфере,
    • создается sync event enqueued.
Ключевой эффект: если данные в карточке изменились (и hash изменился), создается новый пакет, даже когда старый еще в pending/pulled.

3.5 Что пишется в БД и комментарии

При новой записи:

  • onec_counterparty_buffers.status = pending
  • sync event: event_type=enqueued, attempt_no=1, result=success
  • в лид amoCRM добавляется заметка: [1C] Контрагент поставлен в буфер. requestId=..., vin=..., clientType=...

При dedupe:

  • новая запись не создается;
  • добавляется заметка: [1C] Повторный запрос: идентичные данные уже есть в буфере/истории...

4. Поток A: pending (1С забирает пакеты)

GET /api/amocrm/integrations/1c/contacts/pending?limit=50

Правила

  • limit необязательный, по умолчанию 50, диапазон 1..200.
  • Выбираются записи только со статусом pending, сортировка по id ASC.
  • Каждая выданная запись сразу обновляется: status=pulled, pull_attempts += 1, pulled_at=now().
  • На каждую выданную запись создается sync event pulled.

Формат ответа

{
  "count": 2,
  "items": [
    { "...payload #1..." },
    { "...payload #2..." }
  ]
}
Если 1С забрала запись, но не отправила result, запись останется в pulled и сама в pending не вернется.

5. Поток A: result callback (1С → API)

POST /api/amocrm/integrations/1c/contacts/result

Входная валидация

Поле Обязательно Правило Комментарий
requestId Да string Ключ связи с буфером
status Да created|found|error Итог обработки в 1С
1cId Нет string Для success-сценариев желательно передавать
1cID, 1cid Нет string Legacy-алиасы, будут нормализованы в 1cId
processedAt Нет string Парсится через Carbon; лучше ISO-8601 с timezone
vin Нет string Хранится только в payload callback-лога, на матчинг не влияет
error Нет string Текст ошибки для status=error

Что делает обработчик

  1. Ищет запись буфера по request_id. Если не нашел: 404 requestId not found.
  2. Определяет успех: status in [created, found].
  3. Обновляет буфер:
    • успех → status=processed,
    • ошибка → status=error,
    • пишет onec_counterparty_id, onec_processing_status, last_error, processed_at.
  4. Создает sync event callback.
  5. Если успех и есть 1cId: обновляет кастомное поле лида onec_counterparty_id (ID 986431).
  6. Пишет комментарий в лид с деталями callback.

Пример callback success

{
  "requestId": "req_jvqa7tml46vk",
  "status": "created",
  "1cId": "1c_ctr_987654",
  "processedAt": "2026-03-20T14:25:00+10:00"
}

Пример callback error

{
  "requestId": "req_jvqa7tml46vk",
  "status": "error",
  "error": "ИНН не прошел проверку в 1С"
}

8. Логирование и наблюдаемость

Для всех запросов под middleware log.amo.requests пишется JSON-лог в файл:

storage/logs/amo_requests.log

Смотреть в браузере можно через:

GET /api/amocrm/logs?limit=100

Что логируется

  • метод, URL, path, IP, User-Agent, duration, response_status, exception;
  • query, payload, raw_body (с маскированием чувствительных полей).

Что маскируется

Ключи, содержащие: secret, token, authorization, api_key, password.

9. Пошаговое E2E-тестирование

Минимальные предпосылки

  1. Запущен avterminal-api и применены миграции.
  2. В .env заполнены amoCRM-параметры и ONEC_AMO_WEBHOOK_SECRET.
  3. Есть тестовая сделка в amoCRM с доступным leadId.

9.1 Быстрый тест через 1c-mock

1) Получить токен для вызова защищенных endpoint-ов:

curl -sS -X POST http://api-avto-terminal.ru/api/setup-test-user

Из ответа взять token и сохранить:

export ONEC_TEST_TOKEN="...token..."

2) Поставить сделку в буфер:

curl -sS -X POST "http://api-avto-terminal.ru/api/amocrm/deals/contract-ready?secret=YOUR_SECRET" \
  -H "Content-Type: application/json" \
  -d '{"dealId":25147637}'

3) Эмулировать работу 1С полностью (pull + callback):

python3 1c-mock/simulate_1c.py flow --limit 50 --status created

4) Проверить логи:

http://api-avto-terminal.ru/api/amocrm/logs?limit=100

9.3 Матрица тест-кейсов (что обязательно прогнать)

ID Кейс Шаги Ожидаемый результат
T1 Happy path individual contract-ready → pending → result(created) Буфер: processed, поле 986431 заполнено, комментарии в лиде есть
T2 Happy path legal Сделка с компанией → полный flow В payload есть блок реквизитов юрлица
T3 Dedupe без изменений Два раза contract-ready без изменения карточки 2-й ответ: already_buffered, requestId тот же
T4 Новая версия данных Изменить поле (например, ИНН) и снова contract-ready Новый requestId, новая запись pending
T5 Не та стадия Передать pipeline/status не из "Договор" ignored, запись в буфер не создается
T6 Неверный secret contract-ready с неправильным секретом 403 Invalid webhook secret
T7 requestId не найден result с несуществующим requestId 404 requestId not found
T8 Legacy ключ 1cID result с 1cID вместо 1cId Обрабатывается успешно
T9 created без 1cId result(status=created, без 1cId) Буфер: processed, но поле 986431 не обновляется
T10 Проверка limit pending с limit=0 или limit=999 422 Validation failed

10. Частые причины проблем

Симптом Проверить в первую очередь Где смотреть
contract-ready всегда 403 Неверный secret или не тот способ передачи .env, /api/amocrm/logs
pending пустой, хотя ожидали записи Событие могло быть ignored, либо dedupe вернул старый requestId Комментарии лида, таблица buffers, логи

500 на result с processedAt Непарсируемый формат даты Логи приложения и запроса

12. Чеклист готовности к прод

  1. Заданы ONEC_AMO_WEBHOOK_SECRET, ONEC_CONTRACT_PIPELINE_ID, ONEC_CONTRACT_STATUS_ID.
  2. 1С умеет работать с одиночным callback-объектом и всегда отправляет result.
  3. Пройдены T1..T10 без критических дефектов.
  4. Операционный доступ к /api/amocrm/logs согласован (или закрыт по политике безопасности).
  5. Согласовано, как обрабатывать зависшие pulled записи (регламент retries/manual replay).
