Интеграция amoCRM ↔ 1С: фактическая реализация

Документ для программиста 1С. Описывает API-контракт, поведение методов и отклонения от исходного ТЗ.

0. Базовый URL

http://api-avto-terminal.ru

Все URL ниже указаны относительно этого домена.

1. Общая схема

Реализован flow “контрагенты для 1С” в формате буфер + pull + callback:

  1. Источник события вызывает endpoint /api/amocrm/deals/contract-ready.
  2. API формирует payload контрагента и кладет запись в буфер SQLite.
  3. 1С забирает буфер через GET /api/amocrm/integrations/1c/contacts/pending.
  4. После обработки 1С отправляет результат через POST /api/amocrm/integrations/1c/contacts/result.

2. Endpoint’ы и авторизация

Метод URL Кто вызывает Авторизация Назначение
POST http://api-avto-terminal.ru/api/amocrm/deals/contract-ready?secret=... источник события секрет в query/header Поставить контрагента в буфер на отдачу в 1С
GET http://api-avto-terminal.ru/api/amocrm/integrations/1c/contacts/pending?limit=50 1С Bearer (Sanctum) Забрать pending-контрагентов
POST http://api-avto-terminal.ru/api/amocrm/integrations/1c/contacts/result 1С Bearer (Sanctum) Передать результат обработки в 1С
GET http://api-avto-terminal.ru/api/amocrm/logs оператор/разработчик без авторизации Просмотр лога входящих запросов

Секрет webhook задается в .env:

ONEC_AMO_WEBHOOK_SECRET=...

Допустимые способы передачи секрета в webhook:

  • query-параметр ?secret=...
  • header X-Amo-Webhook-Secret или X-Webhook-Secret

3. Источники запуска постановки в буфер

На текущий момент поддержано два источника запуска contract-ready:

  • автоматический триггер;
  • ручной триггер.

Оба источника приводят к одинаковому результату на стороне API: постановка (или схлопывание) записи в буфере.

4. Логика постановки в буфер (важно)

  • Если по этому же lead_id уже есть активная запись в статусе pending или pulled, новая запись не создается (схлопывание дублей).
  • Если payload уже был обработан ранее (processed) и совпал по хешу, новая запись тоже не создается.
  • В лид пишутся сервисные комментарии:
    • постановка в буфер;
    • повторный запрос (already buffered/already processed);
    • итог callback от 1С.

5. Формат, который 1С получает в pending

Пример payload одного элемента items[]:

{
  "vin": "TEST020",
  "dealId": 25147637,
  "clientType": "legal | individual",
  "clientTypeRaw": "Юр. лицо | Физ. лицо",
  "client": {
    "...": "набор полей зависит от clientType"
  },
  "deal": {
    "dealNumber": "25147637"
  },
  "_meta": {
    "contactId": 29639483,
    "companyId": 29639633
  },
  "requestId": "req_xxxxxxxxxxxx"
}

5.1 Для физлица (clientType=individual)

  • fullName, phone
  • passportSeries, passportNumber, passportIssuedBy, passportIssueDate, passportDepartmentCode
  • registrationAddress, inn, snils

5.2 Для юрлица (clientType=legal)

  • name, phone
  • inn, kpp
  • legalAddress
  • rs, bik

6. Формат callback от 1С

POST /api/amocrm/integrations/1c/contacts/result

{
  "requestId": "req_xxxxxxxxxxxx",
  "status": "created | found | error",
  "1cId": "optional for created/found",
  "vin": "optional",
  "processedAt": "ISO datetime",
  "error": "text for error"
}

Эффект callback:

  • буферная запись получает статус processed (или error);
  • при created/found и наличии 1cId обновляется поле лида 986431;
  • в лид добавляется комментарий о результате синхронизации.

7. Отклонения от исходного ТЗ (фактическое состояние)

  1. Статус/дата/ошибка синхронизации пишутся в комментарии лида, а не в отдельные custom fields.
  2. Поле ID контрагента 1С реализовано в лиде (986431) — соответствует согласованному решению.
  3. Номер договора в текущем контракте не передается (поле исключено по согласованию).
  4. Хеш payload хранится на сервере в SQLite.
  5. Интеграция с 1С реализована pull-моделью (pending + result), не прямым push в 1С.
  6. Логи входящих запросов доступны через открытый endpoint /api/amocrm/logs (для отладки).

8. Важные наблюдения с тестов

  • Подтверждено, что оба метода запуска реально приходят и обрабатываются.
  • Ранее были дубли при “этап + кнопка”, это исправлено схлопыванием активной записи по lead_id.
  • В тестах по юрлицу встречался кейс, когда в payload companyId=null и реквизиты юрлица пустые. Это означает, что у сделки не всегда поднимается связанная компания через API-отношение; нужно учитывать на стороне 1С обработку неполных данных.

9. Рекомендации для 1С-разработчика

  • Обрабатывать requestId как идемпотентный ключ операции.
  • Принимать неполные payload (часть полей может быть null / пустой строкой).
  • После обработки обязательно отправлять result, иначе запись останется в pulled.
  • Разделять логику на created (создан новый контрагент), found (найден существующий), error (ошибка).
Примечание по безопасности: в отладочных логах сейчас может отображаться secret из query-параметра. Для продакшена желательно маскировать/не логировать secret.
