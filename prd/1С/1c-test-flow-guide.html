Тестовый поток 1С

Инструкция для разработчика 1С по работе с отдельной тестовой очередью контрагентов.

Базовый адрес: http://api-avto-terminal.ru

Назначение: протестировать обмен с 1С на реальных данных из боевой amoCRM, не забирая боевую очередь и не меняя данные в карточках amoCRM.

1. Коротко о схеме

Тестовый поток использует боевую amoCRM только как источник данных для чтения. После запуска тестового endpoint сервер собирает payload по сделке и кладет его в отдельную очередь с признаком environment=test.

Тестовая 1С забирает данные только из test-очереди и отправляет callback только в test-endpoint. Боевой поток остается на старых URL и работает с environment=production.

Контур Постановка в очередь Забор pending Callback result Запись обратно в amoCRM
Боевой /api/amocrm/deals/contract-ready /api/amocrm/integrations/1c/contacts/pending /api/amocrm/integrations/1c/contacts/result Да обновляется 1cId, добавляются комментарии
Тестовый /api/amocrm/deals/contract-ready-test /api/amocrm/integrations/1c-test/contacts/pending /api/amocrm/integrations/1c-test/contacts/result Нет amoCRM не меняется
Главное правило: тестовая 1С не должна ходить на боевые URL /integrations/1c/contacts/*. Для тестов используются только URL с 1c-test.

2. Что нужно настроить

2.1. На стороне API

На сервере должен быть задан отдельный секрет тестового webhook:

ONEC_AMO_WEBHOOK_TEST_SECRET=секрет_для_тестового_запуска

Также нужен отдельный Bearer token Sanctum для тестовой 1С. Его желательно не использовать в боевом контуре.

2.2. На стороне amoCRM

Для ручного теста можно дергать endpoint напрямую по ID сделки. Если нужен тест через кнопку или webhook amoCRM, он должен отправлять запрос именно на:

POST http://api-avto-terminal.ru/api/amocrm/deals/contract-ready-test?secret=секрет_для_тестового_запуска

Payload может быть одним из поддерживаемых вариантов:

  • {"dealId": 123456} - простой JSON для ручного запуска.
  • lead[id] - формат кнопки из карточки.
  • leads[status][0][id] - формат amoCRM webhook смены статуса.

2.3. На стороне тестовой 1С

В настройках тестовой 1С указать:

  • базовый адрес: http://api-avto-terminal.ru;
  • pending URL: /api/amocrm/integrations/1c-test/contacts/pending;
  • result URL: /api/amocrm/integrations/1c-test/contacts/result;
  • HTTP-заголовок: Authorization: Bearer ТЕСТОВЫЙ_TOKEN;
  • HTTP-заголовок: Content-Type: application/json.

3. Жизненный цикл тестовой записи

  1. Оператор или разработчик запускает contract-ready-test по сделке.
  2. API читает сделку, контакт и компанию из боевой amoCRM.
  3. API формирует payload контрагента и создает запись в test-очереди со статусом pending.
  4. Тестовая 1С вызывает GET /pending и получает список элементов.
  5. При выдаче в 1С запись меняет статус на pulled.
  6. Тестовая 1С обрабатывает данные и отправляет POST /result.
  7. API меняет статус записи на processed или error.
Статус записи Значение
pending Запись создана и ждет забора тестовой 1С.
pulled Запись уже отдана тестовой 1С через GET /pending.
processed Тестовая 1С успешно обработала контрагента и вернула created или found.
error Тестовая 1С вернула ошибку обработки.

4. Endpoint постановки в тестовую очередь

Метод: POST

URL: http://api-avto-terminal.ru/api/amocrm/deals/contract-ready-test

Авторизация: отдельный webhook secret. Можно передать одним из способов:

  • query: ?secret=...;
  • body: {"secret":"..."};
  • header: X-Amo-Webhook-Secret: ...;
  • header: X-Webhook-Secret: ....

Минимальный ручной запуск

curl -X POST "http://api-avto-terminal.ru/api/amocrm/deals/contract-ready-test?secret=TEST_SECRET" \
  -H "Content-Type: application/json" \
  -d '{
    "dealId": 123456
  }'

Успешный ответ

{
  "requestId": "req_ab12cd34ef56",
  "status": "queued",
  "environment": "test",
  "source": "json_deal_id"
}

Повторная отправка без изменения карточки

Если payload по сделке уже есть в test-очереди, новая запись не создается. API вернет существующий requestId:

{
  "requestId": "req_ab12cd34ef56",
  "status": "already_buffered",
  "environment": "test",
  "source": "json_deal_id"
}
Дедупликация test и production выполняется отдельно. Наличие такой же карточки в боевой очереди не мешает создать тестовую запись.

6. Как тестовая 1С отправляет результат

Метод: POST

URL: http://api-avto-terminal.ru/api/amocrm/integrations/1c-test/contacts/result

Авторизация: Authorization: Bearer ТЕСТОВЫЙ_TOKEN

Успешное создание или найденный контрагент

curl -X POST "http://api-avto-terminal.ru/api/amocrm/integrations/1c-test/contacts/result" \
  -H "Authorization: Bearer TEST_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "requestId": "req_ab12cd34ef56",
    "status": "created",
    "1cId": "test-1c-counterparty-001",
    "processedAt": "2026-05-30T10:00:00+10:00"
  }'

Поддерживаемые успешные статусы:

  • created - контрагент создан в тестовой 1С;
  • found - контрагент найден в тестовой 1С.

Ошибка обработки

curl -X POST "http://api-avto-terminal.ru/api/amocrm/integrations/1c-test/contacts/result" \
  -H "Authorization: Bearer TEST_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "requestId": "req_ab12cd34ef56",
    "status": "error",
    "error": "ИНН не прошел проверку",
    "processedAt": "2026-05-30T10:05:00+10:00"
  }'

Ответ API

{
  "requestId": "req_ab12cd34ef56",
  "leadId": 123456,
  "status": "processed",
  "environment": "test"
}

Если отправлен status=error, поле status в ответе будет error.

7. Чего ждать от тестового потока

Ожидание Будет? Комментарий
API прочитает реальные данные из боевой amoCRM Да Payload собирается по реальной сделке, контакту и компании.
Тестовая 1С получит отдельную очередь Да Выборка идет только по environment=test.
Тестовый callback изменит статус записи в API Да Запись станет processed или error.
Тестовый callback запишет 1cId в карточку amoCRM Нет Это специально отключено, чтобы не портить боевую amoCRM.
Тестовый поток добавит комментарий в сделку amoCRM Нет Комментарии в amoCRM добавляет только боевой поток.
Тестовая 1С случайно заберет боевую очередь Нет Если используется правильный URL с 1c-test.

8. Ошибки и диагностика

Нет секрета или неверный secret

HTTP Причина
500 На сервере не задан ONEC_AMO_WEBHOOK_TEST_SECRET.
403 Передан пустой или неверный secret.

Не найден dealId

{
  "error": "Validation failed",
  "details": {
    "dealId": ["dealId не найден в payload"]
  }
}

requestId не найден при callback

Если тестовая 1С отправит callback в test-endpoint по боевому requestId, API вернет 404. Это правильное поведение: очереди изолированы.

Посмотреть последние записи

Для диагностики можно открыть:

GET http://api-avto-terminal.ru/api/amocrm/integrations/1c/debug/statuses?environment=test&limit=10

Этот URL покажет последние записи test-очереди: requestId, leadId, vin, статус, дату забора и дату обработки.

9. Рекомендуемый сценарий тестирования

  1. Выбрать реальную сделку в amoCRM, которую можно использовать как источник данных.
  2. Вызвать contract-ready-test с dealId этой сделки.
  3. Убедиться, что ответ содержит status=queued или already_buffered и environment=test.
  4. Вызвать test pending endpoint из тестовой 1С.
  5. Проверить, что в payload пришли ожидаемые поля клиента, машины и договора.
  6. Создать или найти контрагента в тестовой 1С.
  7. Отправить callback в test result endpoint.
  8. Проверить debug endpoint: запись должна стать processed или error.
  9. Проверить карточку amoCRM: поле 1cId и комментарии не должны измениться.

10. Важные ограничения

  • Тестовый поток не является песочницей amoCRM. Он читает реальные данные из боевой amoCRM.
  • Тестовый поток не пишет изменения в amoCRM, но сам факт чтения зависит от доступности amoCRM API.
  • Если тестовая 1С случайно использует боевой pending URL, она будет работать с боевой очередью. Поэтому URL должны быть жестко разделены в настройках.
  • Один GET /pending переводит найденные записи в pulled. Для повторного теста без изменений API вернет already_buffered, а не создаст новую запись.
  • Чтобы создать новую test-запись по той же сделке, нужно изменить данные, которые входят в payload, или очистить/переоткрыть тестовую запись административно.
  • Callback test-контура принимает только created, found или error.

11. Локальный эмулятор 1С

Для ручной проверки можно использовать скрипт из папки 1c-mock.

Забрать test pending

python3 1c-mock/simulate_1c.py \
  --base-url "http://api-avto-terminal.ru" \
  --token "TEST_TOKEN" \
  --environment test \
  pull \
  --limit 50

Полный test flow

python3 1c-mock/simulate_1c.py \
  --base-url "http://api-avto-terminal.ru" \
  --token "TEST_TOKEN" \
  --environment test \
  flow \
  --limit 50 \
  --status created
