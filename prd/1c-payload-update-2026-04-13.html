Отчет по доработке payload для интеграции с 1С

Ниже зафиксированы изменения, внесенные на стороне сервера обмена для отправки данных из amoCRM в 1С.

Статус: изменения внесены и покрыты автотестом.

Что изменено

  • В исходящий payload добавлены реквизиты сделки: номер договора, дата договора, склад, марка и модель.
  • Номер договора теперь передается как ID сделки.
  • Дата договора определяется по типу клиента: для юрлица берется из компании, для физлица из контакта.
  • Для физлица в payload добавлены реквизиты автодилера: название связанной компании и ее ИНН.
  • Поля вынесены в конфиг, чтобы их можно было менять без повторного хардкода по сервису.

Новый маппинг полей

Поле payload Источник ID поля amoCRM Правило
deal.contractNumber Сделка не поле Берется как ID сделки
deal.contractDate Компания 919495 Если клиент Юр. лицо
deal.contractDate Контакт 947713 Если клиент Физ Лицо
deal.warehouse Сделка 969469 Поле списка Склад
deal.brand Сделка 808679 Поле Марка
deal.model Сделка 808675 Поле Модель
client.dealerName Связанная компания сделки название компании Только для Физ Лицо
client.dealerInn Связанная компания сделки 897733 Только для Физ Лицо, если ИНН заполнен

Какие файлы обновлены

  • avterminal-api/config/amocrm.php — добавлены ID новых полей в конфиг.
  • avterminal-api/app/Services/OneC/CounterpartyFlowService.php — расширен сбор payload для 1С.
  • avterminal-api/tests/Feature/CounterpartyFlowServicePayloadTest.php — добавлены тесты для физлица и юрлица.

Что проверено

Выполнен локальный запуск теста:

php artisan test tests/Feature/CounterpartyFlowServicePayloadTest.php

  • Проверен сценарий Физ Лицо: дата договора из контакта, реквизиты автодилера добавляются.
  • Проверен сценарий Юр. лицо: дата договора из компании, поля автодилера не добавляются.

Итог

Сервер обмена готов передавать в 1С расширенный payload по актуальным правилам из amoCRM, не затрагивая текущую очередь, callback и существующую логику dedupe.
